# AI核心概念

# Model(模型)

# 模型的基本概念

模型是处理和生成信息的算法，是通过海量语料进行预训练得到的参数集合，我们可以将其理解为一个经验丰富的问答专家。

从技术角度来说，现代大语言模型都是基于Transformer架构，即通过自回归预测根据已有序列不断循环推测下一个token。模型参数越多，记忆能力和推理能力也就越强，同时也会带来更高的计算成本和部署复杂度。

现代大语言模型参数通常达到数十亿甚至百亿级别，并编码了数据中的模式和规律，使得模型能够做到：

  1. 从海量数据中正确学习语言模式
  2. 将领域知识编码到参数权重中
  3. 基于学习到的知识正确进行逻辑推理

这也就是为什么大语言模型能够很好的根据输入完成推理和生成。

我们举个具体的例子：当用户询问Java面向对象三大特性，模型的工作步骤为：

  1. 将输入文本token化，转为模型能够识别的语言
  2. 通过注意力机制计算token之间的相关性，推导出下一个可能输出的结果
  3. 通过注意力权重进行推理预测，选择概率最大的一个作为输出

# 大语言模型(LLM)

大语言模型则是模型基础之上的增强版，是一个能力更加综合强大的算法，让用户无需为每个任务训练专门的模型，大模型参数越多，记忆能力也就越强，理解能力也就越好，生成的结果也就更专业且符合预期。

这里笔者也给出业界对模型规模的通用分类标准（以Transformer-based语言模型为例）：

分类 参数规模 代表模型 特点
小模型 < 1亿 DistilBERT 适合端侧部署，推理速度快
中等模型 1亿 - 100亿 LLaMA-7B 具备基础推理能力，可本地运行
大模型 100亿 - 500亿 GPT-3(1750亿)、LLaMA-70B 具备涌现能力，性能显著提升
超大模型 > 5000亿 GPT-4、Claude 3 具备强大的推理和生成能力

补充说明：业界通常认为当模型参数规模超过100亿时，会出现"涌现能力"（Emergent Abilities），即模型在训练数据量达到一定规模后，突然展现出在小模型上不具备的推理、规划等能力，具体表现为：

  1. 复杂推理能力：能进行多步骤的复杂逻辑推理问题分解
  2. 上下文学习：通过少量示例学习新任务，无需重新训练
  3. 指令遵循：能够更好的理解和执行指令复杂的自然语言任务

1750亿是GPT-3的参数量，是大语言模型发展史上的里程碑；万亿级参数是目前超大模型的估算门槛。

# 嵌入式模型(Embedding Model)

模型无法从直接理解文本中的语义，需要Embedding技术实现语义层面上的理解和检索。针对非结构化数据（如文本、图像），Embedding模型将转为高维向量，将人类可理解的语义信息编码到数学的向量空间中。

具体来说，Embedding模型核心工作原理是：

  1. 语义编码：将文本映射到稠密的向量空间，确保相似的语义在向量空间距离更近
  2. 向量检索：通过余弦相似度或者欧氏距离计算向量相关性
  3. 降维优化：通过PCA、t-SNE技术将高位向量维度降低，提升模型理解效率，间接提升检索效率。

这使得该模型经过训练后，可以做到：

  • 语义消歧：苹果这个词，在讨论水果和科技公司时，其向量是不同的，不会因为询问苹果手机的问题，而输出关于水果的回答
  • 语义相似度匹配：猫和狗在向量空间的位置比较接近（都是宠物），而猫和汽车的距离则较远
  • 跨语言理解：不同语言表达相同的概念时，对应词汇在向量空间中较为接近，这也就是为什么用不同语言提问相同问题，并制定输出相同语言时，输出的结果是基本一致的。

这种模型大大提升的数据检索的效率，以图书管理系统为例，传统的数据检索需要精确的关键字才能进行知识检索，我们可能需要通过：

  • 书名模糊查询（MySQL like操作）
  • 书本内容模糊查询（es的search）

所以，如果我们不能提供“相对精确”的关键词，例如检索《哈利波特》这本书就必须提供书本名哈利或者小说章节任意准确的关键字，否则就无法检索到这些信息。

而嵌入式模型将这些书籍的信息、内容都转为向量数据并存储到向量数据库中。当用户输入一个小男孩到魔法学校上学的冒险故事，系统会将其转换为向量，通过相似度计算语义得到最相关书籍，从而定位到本案例的**《哈利波特》**。

# Prompt（提示）

提示词是与模型沟通的入口，本质是通过引导大语言模型按照预期目标推理出最相关结果并输出，提示词的质量对最终效果有着显著作用，但具体提升幅度因模型能力和任务差异而定。

从技术角度上来说，提示词通过以下机制影响LLM推理过程：

  • 注意力引导：提示词中的关键词和上下文信息影响LLM注意力权重分配
  • 推理路径规划：明确的指令为LLM提供正确的推理方向，减少不确定性
  • 输出约束：目标设定和约束条件，限制模型的生成空间

好的提示词应具备系统化的设计方法，结合业界主流实践，我们推荐KITE提示词框架，该框架与LLM内部工作机制深度契合：

  • 认知（Knowledge）：通过知识注入引导LLM注意力权重向相关领域知识倾斜，减小无关信息干扰
  • 指令（Instruction）：明确指令为LLM提供推理路径规划，减少任务理解的不确定性
  • 目标（Target）：设定目标约束LLM输出方向，提升结果回答准确性
  • 限制（Edge）：通过约束界定LLM生成空间，确保输出安全合规

# Token(令牌)

模型理解语言的基本单位是token而非文字，token是模型运作的基础单元，输入的文本会被转为token序列，模型完成预测推理后，再将生成结果token再转为词语输出。

token计算通常是使用BPE（Byte Pair Encoding）等词片算法进行tokenize，即将常见的字符组合合并为单个token，一个token可能对应一个词根或词的一部分。例如"I love programming"就会生成["I", " love", " programming"]序列，也就是3个token。而中文"我喜欢编程"也是同理切割为["我","喜欢","编程"]，也是3个token。

同时，不同的模型都有这固定的上下文窗口**（可处理的token数）**，如下表所示，所以GPT-3.5更适用于简单对话，GPT-4 适合长文档分析和复杂逻辑推理，而Claude 3则适用于处理复杂任务以及源代码分析等：

GPT-3.5 4k token 约3000汉字
GPT-4 128k token 约9.6万汉字
Claude 3 200k token 约15万汉字

token数量直接影响模型的计算成本。一方面，token数量越多，模型的推理时间越长,因为Transformer的自注意力机制计算复杂度为O(n²·d)，推理时间随着序列平方增长;

另一方面，Transformer的注意力机制需要 O(n²) 存储空间，token数量越多，上下文占用的内存空间也会随之增大，这也就是各大厂商采用token作为计费单位的原因。

# Tools（工具）调用技术原理详解

模型的能力在发展初期仅支持自然语言的输入和输出，并不具备与外部进行交互的能力。为了增强模型的能力，设计者提出Tool calling(工具调用)或者function calling的概念，让LLM通过约定好的规范让模型与外部世界交互，从而扩展模型的能力。

Tools封装了输入模式及可调用函数，开发者可以工具注册到兼容的模型，当需要调用工具时，完整的工作流程如下：

  1. 模型生成请求：模型根据任务需求，生成符合工具的格式的参数（ 包含工具名和参数）
  2. 客户端工具调用：客户端拦截该请求，完成实际的工具函数调用
  3. 返回结果：将工具执行结果返回模型，模型基于结果继续生成回答

总的来说，tools的能力大体可以分为以下两种：

  1. 信息检索：通过既定的输入模式发起外部调用，例如：web搜索等、数据库查询等，针对既有模型的知识进行增强，该场景常用于RAG(检索信息增强)，例如：检索给定位置当日天气、web搜索、查询数据库特定记录。
  2. 执行操作：此类别的tool可用于发起外部软件系统调用，例如:发送电子邮件、数据库创建记录、触发工作流等。目标是自动化干预原本是需要人工操作的动作或者显示编程，例如：完成表单填写或者编程中基于TDD的自动化测试实现的Java代码编写。

很多人认为tools是模型的一部分，实际上tool是可以由客户端自行编写实现的函数工具，模型只能生成工具的调用请求和参数，并不具备直接调用tool api的能力，这也是一种关键的安全考虑。

# Memory(记忆)技术原理详解

为什么我们在切换模型时继续进行沟通时，新的模型仍能感知并理解对话的上下文？本质上就是memory记忆功能，memory是AI系统实现连续对话的核心技术，由于LLM本身是无状态的，每次调用都是一次完全独立的推理过程，无法通过对话持续学习和适应用户的偏好。想象一样这样一个场景，用户询问：Spring boot有什么优势？然后接着问“”如何使用“，如果没有记忆功能，模型就无法理解"如何使用"指代的是Spring boot这款Java框架。

为了解决这一问题，设计者提出memory存储历史上下文，让模型能够看到之前的对话内容，从而维持会话的连贯性。在技术实现上，spring AI alibaba将记忆存储Graph的状态中，而状态则是通过checkpointer持久化到数据库或内存中，以便可以随时恢复线程。这也就是为什么即使切换模型或者重启应用，模型仍然能够基于之前的上下文完成合理的回答。

记忆机制为LLM提供了外部大脑，通过持久化存储解决LLM无状态限制，使得AI能够真正理解用户的意图演进，实现从简单回答到自然对话的过渡，提升处理问题的准确性和用户体验。

# Agent核心概念扫盲

# AI Agent基本概念

AI Agent（人工智能智能体）是一种能够感知环境，进行决策并执行实际动作的自动化软件系统。它由LLM、规划能力（Planing）、工具调用（Tools）、记忆（memory）所构成的一个综合性智能体。让原本仅仅能针对文本的LLM，具备推理思考并实际操操作外部世界解决问题的能力。

AI agent收到用户提出问题时，执行如下流程：

  1. 分析当前任务状态，结合记忆对上下文进行推理思考，拆解目标并生成思考路径
  2. 选择合适的工具执行具体操作动作
  3. 接收工具反馈结果，将其纳入上下文进行进一步推理
  4. 重复上述动作将问题空间不断被缩小，知道任务完成

这种闭环的反馈机制使得Agent能够不确定中自动完成纠错，确保任务完成的质量。所以，AI agent的核心价值在于将LLM与外部工具执行能力相结合，实现了纯文本处理到实际操作的跨越，为复杂的业务场景提供了智能解决方案。

# Agent与传统编程模式的区别

传统编程模式与agent应用开发在技术理念上存在着显著的差异。传统编程分为硬编码和工作流（workflow）两种方式，都要求开发人员完成需求梳理和任务拆解、流程设计、开发编码、测试上线等完整流程。是由人主导决策的软件工作模式，就传统编程模式而言，硬编码和工作流的区别是：

  1. 硬编码:强调由程序员主动理解业务需求，自行梳理业务流程并完成所有分支和迭代工作，维护成本较高。同时，对于研发人员业务架构、算法要求较高，更适用于相对固定且迭代风险较小的业务场景
  2. 工作流：通过程序员绘制工作流图，主动于产品团队对齐业务语言，流程清晰可视化，能复杂业务流中降低迭代成本和返工概率

agent应用开发代表了更进阶的业务场景解决方案，适用于哪些硬编码或者工作流无法不能无法完全覆盖的场景。例如开发一个线上故障诊断工具，传统编程模式需要开发人员预先定义所有的故障路径和处理分支。而agent通过观测的数据和表象，让程序自行完成分析，构建思考路径，并调用合适工具完成诊断和结果输出：

agent利用LLM强大的推理能力，通过工具调用不断观测数据，进行动态决策，完成推理和结果输出。即时发生业务迭代，例如：故障诊断场景增加，我们也仅仅需要通过维护自然语言的提示词文档或者RAG等语料知识库即可快速完成应用迭代。

# agent现阶段面临的挑战

因为agent工作的本质是将LLM的独立推理和动态决策能力与外部工具的整合，这其中就涉及诸多不可控因素：

  1. 上下文窗口限制：面对相对复杂需要多轮交互的长任务，历史信息会被截断，导致推理质量下降。这源于Transformer架构注意力机制的计算复杂度(或通俗理解为运算量)与序列长度平方成正比，当任务长度超过模型窗口时，早期的会话内容就会被丢弃。
  2. 幻觉问题：LLM本质是预测推理模型，推理过程中依然会存在虚假事实，虽然工具调用可以在一定程度上验证和纠正错误，但LLM推理生成机制从一开始就决定其输出的不确定性。
  3. token经济问题：token为LLM输入输出的基本单位，token大小决定LLM推理时长和内存占用的开销，所以token也成为各大产商的计费单位，多轮迭代的场任务和工具堆叠导致token消耗极高，费用也会随之飙升，所以是实际研发过程中，我们还需要在效果和成本中做好权衡。
  4. 可观测性不足：与传统编码模式不同，agent应用开发大多数是基于自然语言的动态调整，缺乏编程模式那样函数式统一输出，这就比较考验提示词工程师对于提示词工程能力
  5. 规划能力上限：在涉及深度多步骤的推理任务是，LLM规划能力仍有明显瓶颈，因为缺乏任务的全局视角或上下文窗口限制，整个推理决策流程只能保证当前流程最优，而非全局最优，当前最优推理结果不一定是最终结果的正确路径
  6. 安全性问题：agent具备操作外部工具或API调用能力，存在被恶意攻击的风险，例如：提示词注入攻击等

# Spring AI 架构原理

Spring AI是Java系提供的强大的AI框架，其核心优势是它提供了一套完整的AI应用解决方案：

  1. 统一接口抽象：基于适配器模式，提供统一接口调用，使得开发者无需关心底层实现，只需简单配置，即可完成不同AI模型的接入

这种分层机制使得Spring AI既保证的技术上的灵活性，又提供了良好的开发体验。

# ReActAgent扫盲

这里笔者需要特别强调一下chatModel和ReactAgent的区别，按照SAA官网的说法，ReactAgent是LLM和工具的结合，即Reasoning+Acting理念的AI范式框架，在这个范式中，agent会：

  1. 收到用户请求之后，模型是会针对当前给定上下文信息进行推理思考（Reasoning），决定下一步的决策
  2. 选择工具发起行动（Acting)获取答案
  3. 观测（Observation)工具返回结果
  4. 重复1-3，直到完成请求并输出

这种循环使得agent能够：

  1. 将复杂问题拆解为无数子问题
  2. 不断基于中间结果进行动态决策
  3. 处理需要多个工具协作完成的任务
  4. 在不确定的环境中，实现动态纠错和决策优化

ReActAgent基于这套设计理念，当触发流程调用时，通过运行时创建Graph，完成Agent信息处理流程的定义，Graph包含如下节点：

  • Model node（模型节点)：接收请求调用LLM进行推理和决策
  • Tool node（工具节点)：执行工具调用
  • Hooks node（钩子节点）：添加各种hook插入自定义逻辑

对应我们给出call方法底层的实际调用Graph的实现，即位于Agent类下的doInvoke方法，可以看到，其内部通过getAndCompileGraph构建graph之后，将用户输入作为参数唤起整个工作流程的执行：

问题2提出，所有查询天气的用户都是当前系统用户，可以通过id定位到用户的基本信息，需要支持默认查询用户所在城市的天气情况，即：

基于上述思路，我们先给出UserLocationTool即用户位置查询工具的代码示例，可以看到笔者通过Map映射模拟数据库中的数据，一旦agent对该工具发起调用，UserLocationTool就会从ToolContext中定位userId，到数据库查询用户信息并返回。

需要注意的是，当前没有查询到用户信息，工具会抛出运行时异常，结合SAA官网的说法：

// 用户位置工具 - 使用上下文  
public class UserLocationTool implements BiFunction<String, ToolContext, String> {  
  
    //模拟数据库用户所在城市信息  
    private Map<String, String> userInfo = Map.of("1", "福建",  
            "2", "深圳",  
            "3", "上海",  
            "4", "北京");  
  
  
    @Override  
    public String apply(  
            @ToolParam(description = "User query") String query,  
            ToolContext toolContext) {  
        // 从上下文中获取用户信息  
        String userId = "";  
        if (toolContext != null && toolContext.getContext() != null) {  
            //拿到运行配置  
            RunnableConfig runnableConfig = (RunnableConfig) toolContext.getContext().get(AGENT_CONFIG_CONTEXT_KEY);  
            // 拿到元数据 userId            Optional<Object> userIdObjOptional = runnableConfig.metadata("userId");  
            if (userIdObjOptional.isPresent()) {  
                userId = (String) userIdObjOptional.get();  
            }  
        }  
        //判空校验  
        if (userId == null) {  
            throw new RuntimeException("请指定用户信息");  
        }  
        if (!userInfo.containsKey(userId)) {  
            throw new RuntimeException("用户信息不存在");  
        }  
  
  
        return "用户" + userId + "的所在地是" + userInfo.get(userId);  
    }  
}

# 消息裁剪

上文提到，SAA对于记忆都是一次性存储，每次对话时都是直接将对话内容提交给LLM，随着对话的轮数增加，上下文变得内容冗长，就会出现LLM注意分散，触发幻觉（在大量历史消息中编造回答）的问题。

因此，我们就需要考虑对消息进行整理，常见方案多为以下几种：

  • 修剪消息：在调用LLM之前移除指定前后几条消息
  • 删除消息：从graph中永久删除消息
  • 总结消息：总结历史较早的消息生成摘要并替换

结合我们的需求以及笔者对于天气预报agent工作流的调测，一次完整的agent构建，涉及：

  1. 用户问题的prompt
  2. ASSISTANT对于UserLocationTool的调用和响应
  3. ASSISTANT对于WeatherForLocationTool的调用和响应
  4. 最终输出结果

整体来说，一次完成的输出消息数大约是6条，因为天气查询助手上下文不存在依赖关系，所以一次完整的调用和输出后，这些记忆是都可以直接清楚的，所以本示例的最终逻辑是在消息数超过6条后，对应消息进行修剪：

  
@HookPositions({HookPosition.AFTER_MODEL})//模型调用之后处理消息  
@Slf4j  
public class MessageTrimmingHook extends MessagesModelHook {  
  
    private static final int MAX_MESSAGES = 6;  
  
    @Override  
    public String getName() {  
        return "message_trimming";  
    }  
  
  
    @Override  
    public AgentCommand afterModel(List<Message> previousMessages, RunnableConfig config) {  
        log.info("消息管理,当前消息数:{}", previousMessages.size());  
        log.info("消息管理,当前消息:{}", JSONUtil.toJsonStr(previousMessages));  
  
  
        if (previousMessages.size() <= MAX_MESSAGES) {  
            return new AgentCommand(previousMessages);  
        }  
        //超过6条,即完成一次完整的一轮天气查询后,进行消息修剪  
        return new AgentCommand(previousMessages.subList(previousMessages.size() - 2, previousMessages.size()));  
    }  
  
  
}

# RAG增强

大语言模型虽然强大，但由于其训练机制的限制，预训练的语料会在某个时间点冻结，无法动态获取最新的信息，因此，我们需要让agent检索外部知识，使用这些特定的信息来增强LLM的回答。

此时就需要用到RAG（检索增强生成）技术，其工作流程为：

  1. 检索阶段：将用户提出的问题转为向量，检索向量数据库，获取最相关语义的文档
  2. 增强阶段：将检索到的最相关文档作为上下文（context），与用户问题组成prompt
  3. 生成阶段：LLM基于增强后的prompt生成回答

例如，当我们需要查询天气助手的作者时，agent是无法回答这个问题（因为这是动态信息，不在模型训练数据中），对此我们可以通过两阶段RAG的方式，在生产回答之前，调用我们的知识库获取作者信息，在基于作者基于增强提示词，让LLM基于这份提示词进行推理预测返回作者信息。

对于检索信息增强，我们首先需要构建向量存储将自定义的文档注入，让后在模型调用前完成文档加载。所以笔者首先在Spring容器中声明存储向量，将作者信息注入：

// 在 Agent 开始时检索文档（只执行一次）  
@HookPositions({HookPosition.BEFORE_AGENT})  
public class RAGAgentHook extends AgentHook {  
    private final VectorStore vectorStore;  
    private static final int TOP_K = 5;  
    private static final String RAG_CONTEXT_KEY = "rag_context";  
  
    public RAGAgentHook(VectorStore vectorStore) {  
        this.vectorStore = vectorStore;  
    }  
  
    @Override  
    public String getName() {  
        return "rag_agent_hook";  
    }  
  
    @Override  
    public CompletableFuture<Map<String, Object>> beforeAgent(OverAllState state, RunnableConfig config) {  
        // 从状态中提取用户问题  
        Optional<Object> messagesOpt = state.value("messages");  
        if (messagesOpt.isEmpty()) {  
            return CompletableFuture.completedFuture(Map.of());  
        }  
  
        @SuppressWarnings("unchecked")  
        List<org.springframework.ai.chat.messages.Message> messages =  
                (List<org.springframework.ai.chat.messages.Message>) messagesOpt.get();  
  
        // 提取最后一个用户消息作为查询  
        String userQuery = messages.stream()  
                .filter(msg -> msg instanceof org.springframework.ai.chat.messages.UserMessage)  
                .map(msg -> ((org.springframework.ai.chat.messages.UserMessage) msg).getText())  
                .reduce((first, second) -> second) // 获取最后一个  
                .orElse("");  
  
        if (userQuery.isEmpty()) {  
            return CompletableFuture.completedFuture(Map.of());  
        }  
  
        // Step 1: 检索相关文档（只执行一次，在整个 Agent 执行过程中）  
        List<Document> relevantDocs = vectorStore.similaritySearch(  
                org.springframework.ai.vectorstore.SearchRequest.builder()  
                        .query(userQuery)  
                        .topK(TOP_K)  
                        .build()  
        );  
  
        // Step 2: 构建上下文  
        String context = relevantDocs.stream()  
                .map(Document::getText)  
                .collect(Collectors.joining(" "));  
  
        config.metadata().ifPresent(meta -> {  
            meta.put(RAG_CONTEXT_KEY, context);  
        });  
  
        // Step 3: 将检索到的上下文存储到状态中，供后续 ModelInterceptor 使用  
        // 存储到 state 中，ModelInterceptor 可以通过 request.getContext() 访问  
        return CompletableFuture.completedFuture(Map.of());  
    }  
}

public class RAGContextInterceptor extends ModelInterceptor {  
    private static final String RAG_CONTEXT_KEY = "rag_context";  
  
    @Override  
    public ModelResponse interceptModel(ModelRequest request, ModelCallHandler handler) {  
        // 从请求上下文中获取检索到的 RAG 上下文  
        // RAG 上下文在 AgentHook 的 beforeAgent 中已经存储到状态中  
        Map<String, Object> context = request.getContext();  
        String ragContext = (String) context.get(RAG_CONTEXT_KEY);  
  
        if (ragContext == null || ragContext.isEmpty()) {  
            // 如果没有检索到上下文，直接调用处理器  
            return handler.call(request);  
        }  
  
        // 增强 systemPrompt        String enhancedSystemPrompt = String.format("""  
          你是一个有用的助手。基于以下上下文回答问题。  
          如果上下文中没有相关信息，请说明你不知道。                    上下文：  
          %s          """, ragContext);  
  
        // 合并原有的 systemPrompt 和检索到的上下文  
        SystemMessage enhancedSystemMessage;  
        if (request.getSystemMessage() == null) {  
            enhancedSystemMessage = new SystemMessage(enhancedSystemPrompt);  
        } else {  
            enhancedSystemMessage = new SystemMessage(  
                    request.getSystemMessage().getText() + " " + enhancedSystemPrompt  
            );  
        }  
  
        // 创建增强的请求  
        ModelRequest enhancedRequest = ModelRequest.builder(request)  
                .systemMessage(enhancedSystemMessage)  
                .build();  
  
        return handler.call(enhancedRequest);  
    }  
  
    @Override  
    public String getName() {  
        return "rag_context_interceptor";  
    }  
}

            .inputType(String.class)  
            .build();

            .apiKey(apiKey)  
            .build();  
    //基于 dashscope api 创建chatmodel  
    ChatModel chatModel = DashScopeChatModel.builder()  
            .dashScopeApi(dashScopeApi)  
            .defaultOptions(DashScopeChatOptions.builder()  
                    .withModel(DashScopeChatModel.DEFAULT_MODEL_NAME)  
                    .withTemperature(0.5) //控制输出的随机性（0.0-1.0），值越高越有创造性  
                    .withMaxToken(1000) // 最大输出长度 更多参数请参考 ChatModel 适配  
                    .build())  
            .build();  
  
    //生成skill hook  
    SkillRegistry registry = ClasspathSkillRegistry.builder()  
            .classpathPath("skills")  
            .build();  
  
    SkillsAgentHook skillsHook = SkillsAgentHook.builder()  
            .skillRegistry(registry)  
            .build();  
  
    ReactAgent agent = ReactAgent.builder()  
            .name("天气助手")  
            .model(chatModel)  
            .hooks(new MessageTrimmingHook(),  
                    new RAGAgentHook(SpringUtil.getBean(VectorStore.class)),

            .interceptors(new RAGContextInterceptor())  
            .saver(new MemorySaver())//Agent 通过状态自动维护对话历史。使用 MemorySaver 配置持久化存储,默认使用hashmap  
            .build();  
    //将ReactAgent聚合到WeatherAgent并交由Spring容器统一管理  
    WeatherAgent weatherAgent = new WeatherAgent();  
    weatherAgent.setAgent(agent);  
    return weatherAgent;  
  
}

# 我要做个 Agent 开发，我要怎么做？你把步骤和流程给我说理解？

  1. 梳理需求：明确 Agent 要解决什么问题，确定输入输出边界
  2. 定位可变部分：哪些逻辑需要 AI 自由发挥（如理解用户意图、生成回复）
  3. 用 Skill 或提示词描述可变部分：将可变逻辑用 Prompt/Skill 约束，控制 Agent 行为
  4. 不可变部分封装 Tool：明确参数和返回值，让 Agent 在 ReAct 循环中按需调用
  • 例如：查数据库、调 API、发邮件等确定性操作封装成 Tool
  1. 提示词调试追踪：记录 Agent 的推理过程（Thought → Action → Observation），方便调试和优化
  2. Memory 机制：根据场景决定是否需要记忆上下文（短期记忆/长期记忆）

核心思路：Agent = LLM + Tool + ReAct 循环 + Memory
